## 💥 遇到的问题和解决方案

### 问题 1: GitHub 克隆认证失败

**错误信息:**
```
remote: Invalid username or token. Password authentication is not supported for Git operations.
fatal: Authentication failed for 'https://github.com/...'
```

**原因:** GitHub 已不再支持密码认证

**解决方案:**
- 将 GitHub 仓库设为 Public（公开）
- 公开仓库可以直接克隆，无需认证

---

### 问题 2: pip 安装依赖失败

**错误信息:**
```
error: externally-managed-environment
```

**原因:** Ubuntu 24.04 使用 PEP 668 规范，不允许直接用 pip 安装到系统环境

**解决方案:**
- 创建 Python 虚拟环境: `python3 -m venv venv`
- 激活虚拟环境: `source venv/bin/activate`
- 在虚拟环境中安装依赖

---

### 问题 3: venv 创建失败

**错误信息:**
```
The virtual environment was not created successfully because ensurepip is not available
```

**原因:** Ubuntu 默认未安装 python3-venv 包

**解决方案:**
```bash
sudo apt install python3.12-venv -y
```

---

### 问题 4: 依赖版本冲突

**错误信息:**
```
ERROR: Cannot install pydantic==2.5.0 because these package versions have conflicting dependencies.
chromadb 0.4.22 depends on pydantic>=1.9
ollama 0.4.0 depends on pydantic>=2.9.0 and <3.0.0
```

**原因:** requirements.txt 指定的 pydantic 2.5.0 太旧，不满足 ollama 0.4.0 的要求

**解决方案:**
- 修改 requirements.txt: `pydantic==2.5.0` → `pydantic>=2.9.0`
- 提交修改并推送到 GitHub
- 在服务器上重新拉取代码
- 重新安装依赖

---

### 问题 5: chroma-hnswlib 编译失败

**错误信息:**
```
ERROR: Failed building wheel for chroma-hnswlib
Failed to build chroma-hnswlib
```

**原因:** 缺少 C++ 编译工具链

**解决方案:**
```bash
sudo apt install build-essential python3-dev -y
```

---

### 问题 6: 防火墙配置

**检查结果:** `ufw status inactive`

**原因:** 服务器防火墙未启用

**解决方案:**
- 服务器层面无需配置（防火墙未启用）
- 需要在腾讯云控制台配置安全组规则
- 添加规则: TCP:8501, 允许, 来源 0.0.0.0/0

## 🎯 技术决策记录

### 决策 1: 使用量化模型 qwen2.5:0.5b

**背景:** 服务器内存有限（3.6GB 总计，约 2.6GB 可用）

**选项对比:**
- llama3:8b: 需要 4.5-5GB 内存 → 会导致 OOM
- qwen2.5:0.5b: 仅需 397MB → 完美适配 ✅

**选择:** qwen2.5:0.5b（中文优化，体积小）

**结果:** 总内存占用 ~871MB，剩余 1.5GB，资源充足

---

### 决策 2: 使用公开仓库

**背景:** GitHub 不再支持密码认证

**选项:**
- A: 创建 Personal Access Token（复杂，需要管理 token）
- B: 将仓库设为公开（简单，无认证问题）

**选择:** 方案 B（公开仓库）

**理由:**
- 学习项目，无敏感信息
- 公开仓库可以直接克隆
- 简化部署流程

---

### 决策 3: 进程管理方式

**当前状态:**
- Ollama: 系统服务（systemd），自动启动
- Streamlit: 手动启动，占用终端

**潜在问题:** Streamlit 进程可能在终端关闭后停止

**优化建议（未实施）:**
- 使用 PM2 或 nohup 管理 Streamlit 进程
- 配置自动重启
- 开机自启

**优先级:** 中（当前手动启动可接受）

## 🎓 学习收获

### 1. Ubuntu 服务器配置

**学到的知识:**
- PEP 668 规范（externally-managed-environment）
- 虚拟环境的必要性
- 系统服务管理（systemd）

**经验:**
- Python 3.12 默认不包含 venv，需要单独安装
- Ubuntu 防火墙默认未启用
- 腾讯云安全组是独立于系统防火墙的

---

### 2. 依赖管理

**问题:** pydantic 版本冲突

**解决过程:**
1. 理解错误信息：ollama 需要 pydantic>=2.9.0
2. 修改 requirements.txt
3. 使用 Git 同步代码
4. 重新安装

**经验:**
- 依赖冲突需要仔细阅读错误信息
- Git 是同步代码的好工具
- 版本锁定（==）可能导致问题，范围约束（>=）更灵活

---

### 3. 内存管理

**关键发现:**
- 模型大小对内存影响巨大
- 量化模型可以显著减少内存占用
- CPU 模式下推理速度可接受

**计算:**
- Ollama 服务: 45MB
- qwen2.5:0.5b: ~400MB
- nomic-embed-text: ~200MB
- Python + 依赖: ~200MB
- 向量数据库: ~50MB
- **总计: ~900MB**

---

### 4. 云服务配置

**腾讯云 vs 阿里云:**
- 安全组配置方式略有不同
- 都需要开放端口才能公网访问
- 内网服务端口不需要在安全组开放

**注意:**
- Streamlit 默认端口 8501
- Ollama 默认端口 11434（内网，不开放）
