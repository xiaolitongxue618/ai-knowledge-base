## 实现的功能模块

### 1. 元数据存储（SQLite）

**为什么需要：**

文档上传后需要记录状态。一个文档从上传到可用，会经历几个阶段：
- indexing（正在处理）
- active（可用）
- failed（失败）

如果不在数据库里记录，怎么知道某个文档处理到哪一步了？

**实现要点：**

- 用 SQLite 存储，轻量够用
- 每个文档有唯一的 source_id（文件的 MD5 哈希）
- 这样可以避免重复上传同一个文件

**踩过的坑：**

内存模式（`:memory:`）下的数据库连接问题。

一开始代码是这样的：
```python
def get_document(self, source_id):
    conn = sqlite3.connect(":memory:")
    cursor = conn.cursor()
    cursor.execute("SELECT ...")
```

每次调用 `get_document` 都创建新连接，结果发现表不存在！

**原因：** `:memory:` 模式下，每个新连接都是独立的空数据库。

**解决：**
```python
def __init__(self, db_path):
    self.is_memory_mode = (db_path == ":memory:")
    if self.is_memory_mode:
        # 内存模式要保持连接打开
        self.conn = sqlite3.connect(":memory:")
        self._init_db()
    else:
        # 文件模式每次新建连接
        self._init_db()

def _get_connection(self):
    if self.is_memory_mode:
        return self.conn
    else:
        return sqlite3.connect(self.db_path)
```

这个小坑卡了我半小时。

### 2. 文档解析器

**支持格式：**
- PDF（用 pypdf）
- Markdown（原生读取）
- 纯文本（UTF-8/GBK 兼容）

**实现细节：**

Markdown 解析有个特殊处理：提取标题结构。

为什么要这样做？因为标题是文档的骨架。按标题分块，每个块都有语义上下文，比纯按字数切分要合理得多。

```python
def _parse_markdown(self, file_path):
    with open(file_path, 'r', encoding='utf-8') as f:
        content = f.read()

    # 提取标题结构
    structure = []
    for line in content.split('\n'):
        if line.startswith('#'):
            level = len(line) - len(line.lstrip('#'))
            title = line.lstrip('#').strip()
            structure.append({
                'level': level,
                'title': title
            })

    return content, {'type': 'md', 'structure': structure}
```

这样分块时就知道每个块属于哪个标题下了。

### 3. 文本分块器

**为什么要分块：**

LLM 有上下文长度限制（比如 4K、8K token）。一个几十页的 PDF 肯定塞不进去，所以要先切成小块。

**分块策略：**

- **Markdown：按标题结构分块**
  - 每个 #、##、### 开始新块
  - 这样语义完整，适合知识检索

- **PDF/TXT：按字符数分块**
  - 优先在段落边界（`\n\n`）切分
  - 其次在句子边界（`。`）切分
  - 实在不行再强制切分

**重叠参数（overlap）：**

这是个重要的细节。

假设 chunk_size=500，overlap=50，那么：
- 第 1 块：0-500 字符
- 第 2 块：450-950 字符（重叠 50）
- 第 3 块：900-1400 字符

为什么要重叠？因为如果正好在句尾或段落处切分，可能会丢失上下文信息。

### 4. 向量存储（ChromaDB）

**为什么需要向量数据库：**

用户问一个问题，我们需要在所有文档块里找到最相关的几块。

如果用关键词匹配（like 搜索），效果很差。比如问"人工智能的应用"，文档里写的是"AI的用途"，关键词匹配就找不到。

向量检索的思路是：
1. 把文本转成向量（一串数字）
2. 把问题也转成向量
3. 计算两个向量的相似度
4. 返回最相似的几个块

**ChromaDB 选型：**

- 轻量，本地运行
- 支持内存模式和持久化
- Python API 简单

**实现：**

```python
def similarity_search(self, query_embedding, top_k=3):
    results = self.collection.query(
        query_embeddings=[query_embedding],
        n_results=top_k
    )

    # 转换为统一格式
    return [
        {
            'text': results['documents'][0][i],
            'metadata': results['metadatas'][0][i],
            'similarity': ... # 计算相似度分数
        }
        for i in range(len(results['documents'][0]))
    ]
```

### 6. RAG 完整链路

**这是核心模块，把所有组件串起来。**

**add_document()** - 文档摄入流程：

```python
def add_document(self, file_path):
    # 1. 计算 MD5（去重）
    source_id = self.document_processor.calculate_md5(file_path)

    # 2. 检查是否已存在
    existing = self.metadata_store.get_document(source_id)

    # 3. 创建记录（状态：indexing）
    self.metadata_store.create_document(source_id, ...)

    # 4. 解析文档
    text, metadata = self.document_processor.load_document(file_path)

    # 5. 分块
    chunks = self.text_splitter.split(text, metadata['type'])

    # 6. 向量化（调用 Ollama）
    embeddings = [self.llm_client.embed(chunk) for chunk in chunks]

    # 7. 存储到向量库
    self.vector_store.add_documents(texts, embeddings, ...)

    # 8. 更新状态（indexing -> active）
    self.metadata_store.update_status(source_id, DocumentStatus.ACTIVE)
```

整个流程有完整的状态机：
- 成功 → active
- 失败 → failed（记录错误信息）

**ask()** - 问答流程：

```python
def ask(self, question):
    # 1. 问题向量化
    question_embedding = self.llm_client.embed(question)

    # 2. 检索相关文档
    results = self.vector_store.similarity_search(
        question_embedding,
        top_k=3
    )

    # 3. 过滤低相似度结果
    filtered = [r for r in results if r['similarity'] >= 0.3]

    # 4. 构建上下文
    context = "\n".join([r['text'] for r in filtered])

    # 5. 构建提示词
    prompt = f"""
    参考信息：
    {context}

    用户问题：{question}

    请基于参考信息回答问题。
    """

    # 6. LLM 生成答案
    answer = self.llm_client.generate(prompt)

    return {
        'answer': answer,
        'sources': filtered
    }
```

## 踩过的坑

### 1. Windows 终端编码问题

**现象：**

测试文件里用了 emoji（✅、❌），Windows 终端报错：
```
UnicodeEncodeError: 'gbk' codec can't encode character
```

**解决：**

把 emoji 全部替换成纯文本标记：
```python
# 之前
print("✅ 测试通过")

# 之后
print("[OK] 测试通过")
```

虽然不直观，但至少不会报错。

或者设置终端编码为 UTF-8：
```bash
set PYTHONIOENCODING=utf-8
```

### 2. 拼写错误

**现象：**

RAG 链路里有这么一行：
```python
chunks = self.text_spliter.split(text, metadata['type'])
```

运行时报错：`'RAGChain' object has no attribute 'text_spliter'`

**原因：**

初始化时写的是 `self.text_splitter`，调用时写成了 `self.text_spliter`。

这种拼写错误很隐蔽，IDE 不会提示，只有运行时才会发现。

**解决：**

仔细检查，统一命名：
```python
self.text_splitter = TextSplitter(...)
chunks = self.text_splitter.split(text, metadata['type'])
```

### 3. 依赖版本问题

**现象：**

requirements.txt 里写的是：
```
pypdf==2023.9.18
```

安装时报错：找不到这个版本。

**原因：**

pypdf 的版本号不是日期格式，而是 3.x.x 格式。

**解决：**

改成：
```
pypdf==3.17.4
```

同理，numpy 的老版本（1.24.3）在 Python 3.14 上编译失败，需要用新版本：
```
numpy>=1.26.0
```

## 心得体会

### 1. 测试驱动很重要

这次开发基本是"先写测试，再写代码"的模式。

好处：
- 知道要实现什么（测试就是需求）
- 即时反馈，代码一改完就知道对不对
- 回归保护，后续改动不会破坏已有功能

如果没有测试，我肯定不会发现内存模式数据库连接的问题。

### 2. 日志很关键

开发时日志帮我快速定位问题：

```python
logger.info(f"开始解析文档: {file_path}")
logger.info(f"文档分块完成: {len(chunks)} 个块")
logger.error(f"向量化失败: {e}")
```

遇到问题看日志，比打断点调试快多了。

### 3. 状态机要真正落地

设计文档里画了状态机图是一回事，代码里真正实现是另一回事。

这次的状态机是真正落地的：
- 创建时 → indexing
- 成功 → active
- 失败 → failed

每个状态转换都有明确的代码实现和日志记录。

这样系统出问题时，能清楚地知道文档卡在哪个状态。

### 4. 模块化设计的优势

整个系统分成几个模块：
- storage（存储层）
- processors（处理层）
- llm（LLM 客户端）
- rag（业务层）

好处：
- 职责清晰，便于调试
- 可以独立测试每个模块
- 后续替换某个模块很容易（比如换个向量数据库）
